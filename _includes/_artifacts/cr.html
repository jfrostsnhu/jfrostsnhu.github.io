Code reviews, also known as peer reviews, are a vital step in the software development lifecycle. They provide a systematic assessment of code to improve quality, identify bugs, enhance security, and ensure adherence to coding standards before merging changes into a production environment. By incorporating code reviews into a team's workflow, developers can share knowledge, maintain compliance, and foster collaboration. Code reviews not only help catch issues early in the development process but also facilitate continuous learning among team members, making them an essential practice in software engineering.

As part of my Capstone project, I conducted a code review of an Inventory Management Android application that I developed last year. The purpose of this review was to evaluate the current implementation, identify areas for improvement, and determine the best approach for enhancing security, efficiency, and maintainability. To systematically assess the application, I focused on five key areas including functionality, structure, security, database interactions, and performance. I began with a user experience assessment, testing login, inventory management, and notifications not only to demonstrate the app but also to evaluate how well the core features functioned. Next, I conducted a code structure review identifying areas that needed improved modularity and readability. My security audit revealed vulnerabilities such as storing user roles in plaintext SharedPreferences and a lack of input validation for all fields. The database review highlighted outdated query methods, particularly the use of the Cursor API instead of a modern ORM as well as missing CRUD operations for user management. Lastly, the performance analysis identified inefficiencies in linear searches for inventory tracking which could be optimized using better data structures. While the original artifact successfully implements its core functionality, several areas require optimization, particularly expanding CRUD operations, improving security mechanisms, and refining database interactions.

Summary of Key Findings:

  • Methods like onCreate() violate the Single Responsibility Principle and need refactoring.
  • Consistent naming conventions used, but long lines reduce readability.
  • User roles stored in plaintext SharedPreferences, making them vulnerable to tampering.
  • No input validation for email format or password strength.
  • Lacks encryption for sensitive data.
  • Uses outdated Cursor API instead of a modern ORM such as Room for improved database efficiency.
  • Limited user management (no ability to add, edit, or delete users).
  • Inefficient inventory processing uses linear searches for low-stock alerts.
  • Hardcoded strings reduce maintainability.
